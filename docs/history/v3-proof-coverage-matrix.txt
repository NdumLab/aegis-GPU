V3 Proof Coverage Matrix

Purpose
- Define the next proof-expansion phase after v3 phase 2.
- Separate high-value browser proof from lighter smoke coverage so validation stays disciplined and affordable.

Current Proven Scope
- Adaptive guidance:
  - `study_progress_empty`
- State-aware Ask Aegis:
  - `ask_aegis_main`
- Detached coach behavior:
  - `ask_aegis_detached`
- Branch consequence flow:
  - `ecc_bad`
  - `nvlink_bad`
  - `storage_warn`

Validation Levels
- Browser proof:
  - Use for branch routing, detached behavior, analytics state transitions, and state-aware interaction.
- Frontend smoke only:
  - Use for static rendering, text presence, and simple deterministic sections that do not depend on multi-step browser behavior.
- Backend unittest smoke:
  - Keep for API/auth/regression checks, not frontend teaching-flow proof.

Best/Warn/Bad Combinations
- Browser proof should aim for:
  - one `best` path per major domain
  - one `warn` path per major domain
  - one `bad` path per major domain

Recommended Phase Order
1. Add browser proof for analytics recommendation transitions.
2. Add `best` and `bad` proof for `nccl_fallback`, `cuda_stack`, and `k8s`.
3. Add `best` and `bad` proof for `slurm`, `allreduce`, and `ib_fabric`.
4. Add clean vs compromised completion proof.
5. Add remaining representative browser proof for `roce`, `mig`, and more Ask Aegis intents.

Stop Rule
- Stop the broader-proof phase when:
  - each major domain has at least one `best` and one degraded browser-proven path
  - analytics recommendation transitions are browser-proven
  - clean vs compromised completion is browser-proven
  - incremental proof additions are no longer uncovering new classes of issues

What This Phase Would Produce
- larger targeted browser proof matrix

- updated evidence summaries
- a more credible release/pilot validation package
